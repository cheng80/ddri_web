# DDRI 시스템 설계

작성일: 2026-03-19  
목적: 현재 합의된 운영 방향을 기준으로 웹서비스의 시스템 구조, 데이터 흐름, 최소 영속 계층을 다시 정의한다.

## 1. 시스템 방향

현재 웹서비스는 다음 전제를 기준으로 설계한다.

- 로그인 없는 공개 모니터링 웹
- 입력형 서비스가 아니라 조회형 서비스
- 실시간 동적 데이터는 외부 API를 기준 진실로 본다
- 정적 마스터 데이터는 웹/서버에 선탑재할 수 있다
- 예측 모델은 서버에서 로드하고 추론한다
- DB는 기본 저장소가 아니라 선택적 영속 계층으로 본다

## 2. 상위 아키텍처

```text
Flutter Web
  -> FastAPI
      -> 로컬 마스터 JSON
      -> 예측 모델(joblib)
      -> 외부 재고 API
      -> 외부 날씨 API
      -> 선택적 예측 로그 저장소
```

### 각 계층 역할

- Flutter Web
  - 화면 렌더링
  - 사용자/관리자 조회 요청
  - 선탑재 마스터 데이터 보관 가능
- FastAPI
  - 외부 API 호출
  - 피처 생성
  - 예측 실행
  - 응답 조합
- 로컬 마스터 데이터
  - 대여소 메타
  - 군집
  - 거치대 수
  - 정책 기본값
- 외부 API
  - 실시간 재고
  - 날씨 또는 예보
- 선택적 영속 계층
  - 예측 로그만 저장

## 3. 데이터 계층 재정의

### 3.1 로컬에 두는 데이터

현재 기준으로 아래 데이터는 DB보다 로컬 JSON 또는 메모리 로드 구조가 적합하다.

- 대여소 마스터
- 스테이션 이름, 주소, 위경도
- 군집 코드
- 거치대 수
- 운영 기본 정책값
- 모델 메타데이터

### 3.2 외부 API에서 받는 데이터

- 실시간 자전거 보유 수
- 운영 상태
- 날씨 또는 예보 정보

이 데이터는 요청 시점 조회를 기본으로 하며, 필수적으로 DB에 적재하지 않는다.

날씨 정보는 현재 두 가지 용도로 사용한다.

- 예측 모델 입력 피처
- 사용자/관리자 화면에 표시하는 주간 날씨 및 선택 시각 예상 날씨

### 3.3 선택적으로 저장하는 데이터

현재 단계에서 영속 저장이 필요하다면 예측 로그만 우선 고려한다.

- 예측 시각
- 목표 시각
- 스테이션 번호
- horizon
- 예측 재고
- 예측 순변화
- 모델 버전

실제값 히스토리는 운영 중 즉시 쌓지 않고, 나중에 공공데이터 포털 공개 자료와 배치 매칭할 수 있다.

## 4. DB 사용 원칙

### 현재 결론

- DB는 없어도 운영 가능하다.
- 다만 예측 로그를 서비스 이력으로 남기려면 최소 영속 저장소는 둘 수 있다.
- 따라서 현재 DB는 필수가 아니라 선택 사항이다.

### DB를 도입하는 경우에도 최소 범위

1. `prediction_logs`
   - 예측 요청/결과 이력
2. 향후 필요 시
   - `policy_configs`
   - `system_sync_logs`

현재는 `stations`, `realtime_station_stock`, `station_demand_forecasts`, `station_risk_snapshots`, `statistics_snapshots`를 모두 DB 테이블로 둘 필요는 없다.

## 5. 최소 ERD 재정의

현재 기준 최소 영속 구조는 아래 수준으로 정의한다.

### 선택 1. 완전 무상태 운영

- DB 없음
- 로컬 마스터 + 외부 API + 모델 추론만 사용

### 선택 2. 예측 로그 저장 운영

```text
prediction_logs
 - id PK
 - prediction_time
 - target_time
 - station_id
 - horizon_hours
 - predicted_stock
 - predicted_net_change
 - model_version
 - source_updated_at
 - created_at
```

이 경우에도 스테이션 마스터는 DB가 아니라 로컬 기준으로 둔다.

## 6. 런타임 데이터 흐름

### 사용자 페이지

1. 웹이 위치/시간 기준 조회 요청
2. 서버가 실시간 재고와 날씨 조회
3. 서버가 로컬 마스터를 결합
4. 서버가 예측 모델 실행
5. 서버가 가까운 대여소 목록 응답 생성

### 관리자 페이지

1. 웹이 기준 시각과 필터를 포함한 조회 요청
2. 서버가 실시간 재고와 예측 결과를 조합
3. 서버가 위험도와 우선순위를 계산
4. 서버가 목록과 예외 스테이션을 함께 응답

## 7. 마스터 데이터 운영 방식

- 1차적으로 웹 또는 서버에 선탑재 가능
- API 수신 가능 시 최신화
- 최신화 실패 시 로컬 데이터로 폴백

버전 번호가 필수는 아니지만, 아래 중 하나는 유지하는 것이 좋다.

- `updated_at`
- `etag`
- `hash`

## 8. 예측 평가 방식

현재는 운영 중 실제값을 바로 저장하지 않는 방식을 우선 허용한다.

### 운영 중 저장

- 예측 로그만 저장

### 사후 평가

- 공공데이터 포털의 공개 이력 데이터 확보
- `station_id + target_time` 기준 매칭
- 예측값과 실제값 비교

## 9. 현재 설계 결론

이 서비스는 DB 중심 시스템이 아니라, 외부 API와 로컬 마스터 중심의 조회형 시스템으로 보는 것이 맞다.

현재 설계 우선순위는 아래와 같다.

1. 화면 설계 정합성
2. 외부 API 연동
3. 예측 런타임 구성
4. 예측 로그 저장 여부 결정
5. 필요 시 최소 DB 추가
