# DDRI API 및 런타임 계약

작성일: 2026-03-19  
목적: 현재 화면과 시스템 설계를 기준으로 API 계약과 예측 런타임 입력/출력 구조를 최소 기준으로 정의한다.

## 1. 기본 원칙

- API는 조회 전용을 기본으로 한다.
- 실시간 재고는 외부 API를 기준 진실로 본다.
- 예측 모델 입력 전체를 프론트가 직접 만들지 않는다.
- 프론트는 최소 입력만 보내고, 서버가 런타임 피처를 구성한다.

## 2. 프론트가 보내는 최소 입력

### 사용자 페이지

- 사용자 위치 또는 주소 좌표
- 기준 시각 또는 목표 시각
- 필요 시 반경, 개수 제한

예시:

```json
{
  "lat": 37.5012,
  "lng": 127.0396,
  "target_datetime": "2026-03-19T18:00:00+09:00",
  "radius_m": 500
}
```

### 관리자 페이지

- 기준 시각
- 긴급 여부
- 행정동 필터
- 정렬 기준

예시:

```json
{
  "base_datetime": "2026-03-19T18:00:00+09:00",
  "urgent_only": true,
  "district_name": "역삼동",
  "sort_by": "risk_score"
}
```

## 3. 서버가 내부에서 생성하는 값

예측 모델이나 후처리 계산에 필요한 대부분의 값은 서버가 내부에서 붙인다.

- `current_stock`
- `hour`, `weekday`, `month`
- 최근 시계열 이력
- rolling 피처
- 날씨 또는 예보
- 스테이션 메타
- 군집
- 모델 버전

즉 프론트 입력과 모델 실제 입력은 다르다.

## 4. 예측 런타임 계약

### 기본 흐름

1. 프론트 요청 수신
2. 서버가 외부 API 조회
3. 서버가 로컬 마스터 로드
4. 서버가 피처 생성
5. 서버가 `joblib` 추론
6. 서버가 화면용 응답 조합

### 현재 예측 방식

- 현재 재고 + 예측 순변화
- 또는 관리 화면용 조합 계산

예시 개념:

```text
predicted_stock = current_stock + predicted_net_change
```

## 5. API 목록

### 사용자 조회

`GET /v1/user/stations/nearby`

용도:
- 지정 위치 기준 근처 대여소 조회
- 현재 재고와 예측 결과를 함께 반환

### 관리자 조회

`GET /v1/admin/stations/risk`

용도:
- 기준 시각의 부족 위험 목록 조회
- 재고, 예측 결과, 위험도, 우선순위를 함께 반환

### 스테이션 마스터

`GET /v1/stations`

용도:
- 화면 렌더링용 스테이션 기본 정보
- 필요 시 로컬 선탑재 데이터와 비교 가능

### 날씨

`GET /v1/weather/*`

용도:
- 예측 피처
- 화면 표시용 주간 날씨
- 선택 날짜/시간 기준 예상 날씨

## 6. 날씨 응답 원칙

날씨는 두 층으로 다룬다.

### 주간 날씨

- 기본적으로 사용자 탭과 관리자 탭에 모두 노출
- 오늘 포함 일 단위 7일 예보
- 날짜, 날씨 상태, 아이콘, 최고/최저 기온 포함

### 선택 시각 날씨

- 사용자가 날짜/시간을 고르면 그 시각 기준 예상 날씨를 별도 표시
- 화면에서는 강조 카드 또는 상세 패널로 표현
- 과거 날짜는 허용하지 않는다
- 선택 가능 범위는 현재 시점부터 7일 이내로 제한한다

예시 응답 구조:

```json
{
  "weekly_forecast": [
    {
      "date": "2026-03-20",
      "weather_type": "맑음",
      "weather_low": 4,
      "weather_high": 13,
      "icon_url": "..."
    }
  ],
  "selected_forecast": {
    "weather_datetime": "2026-03-20T18:00:00+09:00",
    "weather_type": "구름많음",
    "weather_low": 6,
    "weather_high": 14,
    "icon_url": "..."
  }
}
```

## 7. 응답 구성 원칙

### 사용자 응답

- 화면 렌더링에 필요한 값만 반환
- 모델 내부 피처 전체를 노출하지 않음
- 예외 스테이션은 별도 배열로 분리 가능
- 주간 날씨와 선택 시각 날씨를 함께 포함할 수 있다

### 관리자 응답

- 목록 중심
- 재고, 예측값, 차이, 위험도, 우선순위 포함
- 예외 스테이션은 별도 배열
- 기준 시각 판단을 돕는 날씨 정보 포함 가능

## 8. 예외 처리 원칙

- 실시간 API 실패 시 예외 스테이션 또는 예외 응답으로 분리
- 일부 스테이션 데이터가 비어도 전체 응답은 가능한 한 유지
- 폴백 데이터 사용 시 응답에 상태를 명시할 수 있다

예:

```json
{
  "data_source_status": "fallback_master",
  "items": []
}
```

## 9. 예측 로그 저장 계약

저장이 필요하면 아래 정도만 남긴다.

```json
{
  "prediction_time": "2026-03-19T14:00:00+09:00",
  "target_time": "2026-03-19T18:00:00+09:00",
  "station_id": 2328,
  "horizon_hours": 4,
  "predicted_stock": 3.2,
  "predicted_net_change": -2.8,
  "model_version": "stock-horizon-ridge-v1"
}
```

실제값은 운영 중 바로 쓰지 않고, 나중에 공개 데이터와 배치 매칭할 수 있다.

## 10. 현재 결론

- 프론트는 최소 입력만 보낸다.
- 서버가 모델 입력 피처를 조합한다.
- API는 화면 응답 중심으로 단순하게 유지한다.
- DB는 API 계약의 필수 조건이 아니다.
